## Presentations

### 1. Fondations — LLM

> Il ne comprend pas — il complète.

Un modèle entraîné sur des centaines de milliards de mots pour prédire le token suivant. Toute la puissance vient de l'échelle.

- **Paramètres** — poids ajustés à l'entraînement, 7B → 1 000B+. Plus = plus capable, plus coûteux.
- **Fine-tuning** — ré-entraîner sur des données métier pour spécialiser le modèle. Touche aux poids.
- **Inférence** — phase de production : le modèle génère token par token, en streaming.
- **Pré-entraînement** — la phase massive : milliers de GPU, semaines de calcul, web entier digéré.
- **Hallucination** ⚠ — texte plausible mais faux. Toujours sourcer et vérifier les faits.

### 2. Fondations — DeepSeek, le choc open source

> Le game-changer open source.

Janvier 2025 : un labo chinois sort un modèle de raisonnement rival de GPT-4o, entraîné pour une fraction du coût — onde de choc dans tout le secteur. 10–30× moins cher que l'API OpenAI à l'inférence.

- **V3 (Chat & Coding)** — modèle de base, 671B paramètres MoE, comparable à GPT-4o, poids ouverts.
- **R1 (Reasoning)** — entraîné par RL pur, rival de o1, chaîne de pensée visible.
- **MoE (architecture)** — Mixture of Experts : seuls ~37B paramètres actifs sur 671B à chaque token.
- **Coût** — entraînement ~6M$ vs ~100M$ pour GPT-4. Marges sous pression partout.

### 3. Fondations — Token, Contexte, Température

> L'unité de traitement, la mémoire de travail, le curseur créativité/déterminisme.

- **Token** — ~¾ mot anglais, 1 000 tokens ≈ 750 mots. Impact direct sur coût et latence. Tokenisation BPE : un mot fréquent = 1 token, un mot rare = 2-5.
- **Context window** — tout ce que le modèle « voit » en même temps (prompt + historique + outputs). GPT-4o 128k · Claude 3.5 200k · DeepSeek-V3 128k · Gemini 1.5 1M.
- **Température** — échelle 0.0 → 2.0, contrôle l'aléatoire de l'échantillonnage. 0.0 déterministe · 0.7 équilibré · 1.2 créatif · 2.0 aléatoire. Code/SQL/JSON : 0.0–0.3. Écriture : 0.7–1.2.

### 4. Les données — RAG (Retrieval-Augmented Generation)

> Brancher le LLM sur vos documents.

Le modèle ne voit pas vos données. On les indexe, on les récupère par similarité, et on les injecte dans le contexte au moment de répondre.

Pipeline : **Document** (PDF, Word, Notion, web, Confluence…) → **Chunking** (blocs ~500 tokens, overlap 50) → **Embedding** (texte → vecteur sémantique) → **Vector DB** (Pinecone, Chroma, pgvector, Weaviate) → **Retrieval** (similarité cosinus, top-k chunks) → **Réponse** (LLM + contexte = réponse sourcée).

- **Embedding** — deux textes sémantiquement proches ont des vecteurs proches. Base de toute recherche moderne (ex. text-embedding-3-small).
- **Vector DB** — base optimisée pour la recherche par similarité cosinus à grande échelle (ex. Pinecone, pgvector).
- **Chunking** — taille et overlap impactent fortement la qualité de récupération ; à tester sur ses propres docs (ex. 500 tokens, overlap 50).

### 5. Les agents — Anatomie d'un agent IA

> Un agent agit. En boucle.

Agent = LLM + boucle de raisonnement + outils + mémoire. Objectif donné → planifie → agit → observe → s'adapte.

- **LLM** — le cerveau : raisonne sur l'objectif, choisit l'étape suivante.
- **Tools** — les bras : fonctions appelables (web, code, APIs métier).
- **Memory** — le souvenir : court terme (contexte) + long terme (DB).
- **Planning** — la stratégie : Chain-of-Thought, Tree-of-Thought, ReAct.
- **Boucle ReAct (Reason + Act)** — Think (Reason) → Act (Tool) → Observe (Result) → Adapt (Loop).

### 6. Les agents — Tool use

> L'agent ne connaît pas les outils. Il les choisit.

Flux : user → LLM choisit un outil → JSON call → runtime exécute → LLM reçoit le résultat → réponse finale.

- **Recherche & Web** — `web_search()` · `browse_url()` · `news_fetch()` · `wikipedia()`
- **Code & Fichiers** — `python_exec()` · `bash()` · `read_file()` · `write_file()`
- **APIs & Services** — `email.send()` · `calendar.create()` · `slack.post()` · `github.pr()`
- **Données** — `sql_query()` · `csv_read()` · `vector_search()` · `scrape()`

Le pouvoir d'un agent = la qualité de sa boîte à outils.

### 7. L'écosystème — MCP (Model Context Protocol)

> Le « USB-C » des agents IA.

Protocole ouvert d'Anthropic (nov. 2024) : n'importe quel LLM ↔ n'importe quel outil.

- **Host** (l'application) — Claude, GPT-4o, Cursor/Zed, votre app → **MCP Client** ⇄ Protocol ⇄ **MCP Server**.
- **Servers** (les capacités) — `filesystem` (lire/écrire des fichiers locaux), `github` (issues, PRs, code), `postgres` (requêtes SQL), `slack` (messages, canaux), `brave-search` (recherche web), `custom` (votre propre API métier).
- Un serveur MCP expose tools, ressources et prompts de manière standardisée.

### 8. L'écosystème — Skills, System Prompt, Orchestration

> Construire un système agentique : quatre primitives à connaître.

- **Skill / Tool** — capacité encapsulée donnée à un agent (fonction Python, prompt template, API wrappée). L'agent choisit quel skill invoquer. Ex. `search_web(query)` · `send_email(to, subject)`.
- **System Prompt** — instructions persistantes avant la conversation, définissant rôle/limites/ton, invisibles pour l'utilisateur final. Ex. « Tu es un assistant RH. Ne parle pas des salaires. »
- **Orchestrateur** — code/framework qui coordonne plusieurs agents et appels LLM, gère flux, dépendances, agrégation des résultats. Ex. LangChain · LangGraph · CrewAI · n8n · Make.
- **Multi-agent** — plusieurs agents spécialisés collaborent ; un manager délègue à des sous-agents (researcher, coder, reviewer…). Ex. A cherche → B rédige → C valide → D publie.

### 9. L'écosystème — Memory, Grounding, Fine-tuning

> Mémoire d'un agent : quatre étages, des vitesses différentes.

| Type | Analogie | Description |
|---|---|---|
| In-context | RAM (volatile, session) | Tout ce qui tient dans le context window. Rapide, effacé à chaque nouvelle conversation. |
| External DB | SSD (persistant) | Vector DB ou SQL, survit entre sessions, récupéré à la demande via RAG ou tool use. |
| Épisodique | Journal (append-only) | Logs des actions passées et de leurs résultats — apprentissage par l'expérience, debugging des décisions. |
| Sémantique | Poids (gravé à l'entraînement) | Connaissances dans les poids du modèle, modifiable uniquement par fine-tuning. |

- **Grounding** (RAG + Tools) — ancrer le LLM sur des faits réels, réduit les hallucinations sans modifier les poids.
- **Prompt Engineering** (Instructions) — instructions système + few-shot pour guider le comportement, sans toucher aux poids.
- **Fine-tuning** (Poids) — ré-entraîner sur ses données, modifie les poids, coûteux mais utile pour un style spécifique.

### 10. Démo live nº01 — Chatbot RAG sur vos propres données

On branche un LLM générique sur un dossier de PDF métier. La même question, deux réponses radicalement différentes.

- **Stack** — LangChain · Chroma · OpenAI Embeddings (text-embedding-3-small)
- **Sans RAG** — « Je n'ai pas accès à vos documents internes, mais en général… » ⚠ réponse générique, hallucination possible.
- **Avec RAG** — « D'après la page 12 de votre politique RH, le délai est de… » ✓ sourcé, vérifiable, chiffres exacts.
- Commande démo : `chatbot --docs ./policies/ --ask "délai préavis CDI ?"` → retrieving 4 chunks (cosine ≥ 0.78)… sources : politiques-rh.pdf p.12, p.34.
- → Question pour la salle : sur quels documents vous voudriez ça ?

### 11. Démo live nº02 — Agent researcher, ReAct en action

Un input simple. L'agent planifie, cherche, browse, filtre, et rend un rapport markdown — chaque étape du raisonnement est visible.

1. **Input** (user) — « Veille agentic AI cette semaine »
2. **Think** (LLM) — planifie les sources et requêtes
3. **Act** (tool) — `web_search()` · `browse()`
4. **Observe** (LLM) — filtre le bruit, garde 5 sources
5. **Rapport** (output) — markdown sourcé, par thème

- **Stack** — Claude 3.5 Sonnet · Tool use natif · MCP brave-search
- **Self-healing** — l'agent se corrige si une source est inaccessible, pas de plantage, une autre route.
- → Question pour la salle : quelle tâche répétitive vous voudriez déléguer ?

### 12. Démo live nº03 — Build en live avec Claude Code
